14.1 比较访问控制模型

授权主体访问客体的方法取决于 IT 系统采用的控制方法。

14.1.1 比较权限、权利和特权

权限

指授予对客体的访问权限，并明确允许对客体执行的操作。

权利

指对某个客体采取行动的能力。

权利和权限有细微的区别，并不总是被刻意强调。

特权

特权是权利和权限的组合。

14.1.2 理解授权机制

隐式拒绝

访问控制的基本原则是隐式拒绝，大多数授权机制使用隐式拒绝。

除非明确授予主体访问权限，否则，隐式拒绝原则拒绝主体访问客体。

也可以视为默认拒绝。

访问控制矩阵

访问控制矩阵是一种包含主体、客体和分配权限的表格。

访问控制矩阵并非只是一个访问控制列表（ACL），存在多维控制列表。

能力表

能力表（capability table）是另一种确定主体所分配权限的方式。

能力表主要关注的是主体（用户、组或角色），体现主体可以访问的客体信息。

ACL 主要关注客体，体现能够访问某个客体的主体信息。

约束接口

应用程序利用受约束或受限制的接口，限制用户可执行或可查看的内容。

常见方法：

用户无权使用某功能时隐藏该功能。

和 Clark-Wilson 模型 相关联

基于内容的控制

基于内容的访问控制依据客体内容来限制对数据的访问。

例如：数据库视图就是一种基于内容的控制。

基于上下文的控制

基于上下文的访问控制在授予用户权限之前需要用户执行特定的活动。

基于上下文的控制也可以使用日期和时间控件，在允许时间之外访问会被拒绝。

因需可知

确保主体仅能够访问为完成工作任务和工作职能而需要知道的内容。

只给需要的权限。

最小特权

最小特权原则确保主体仅拥有执行工作任务和工作职能所需的权限。

最小特权和因需可知的区别：
最下特权包括在系统上执行操作的权利，也就是可以在系统上执行的能力。

职责分离

确保将敏感职能划分为两个或多个员工共同执行的任务。

通过创建一个检查和制衡系统，防止欺诈和错误的发生。

14.1.3 使用安全策略定义需求

安全策略是一种定义组织安全需求的文档，确定了组织需要保护的资产、安全解决方案以及需要保护的程度。

可以单个文档，也可以多个文档。

安全策略是实现安全需求的指南，也就是方向，而不是具体的实施操作

14.1.4 介绍访问控制模型

自主访问控制 DAC

自主访问控制（DAC）模型的一个关键特征是每个客体都有一个所有者，所有者可以允许或拒绝其他主体的访问。

实例：

微软 Windows 的 NTFS 文件系统使用 DAC 模型。

基于角色的访问控制 RBAC

基于角色的访问控制（RBAC）模型的一个关键特征是角色或组的使用。

不直接向用户分配权限，而是向角色分配权限。

角色通常基于职责功能来定义。

实例：

微软 Windows 操作系统使用组来实现 RBAC 模型。

基于规则的访问控制

基于规则的访问控制模型的一个关键特征是采用适用于所有主体的全局规则。

规则也被称为“限制” 或 “过滤器”。

例如防火墙。

基于属性的访问控制 ABAC

基于属性的访问控制 ABAC 模型的一个关键特征是使用包含多个属性的规则。

ABAC 基于属性的比基于规则的更灵活。

强制访问控制 MAC

强制访问控制模型 MAC 的一个关键特征是使用应用于主体和客体的标签。

基于标签。

MAC 强制访问控制模型被称为基于格子的模型。

基于风险的访问控制

基于风险的访问控制模型在评估风险后授予访问权限。

运用机器学习并根据过去的活动对当前活动做出预测。

14.1.5 自主访问控制

自主访问控制系统，允许客体的所有者、创建者或数据所有者、数据托管员控制和定义对客体的访问。

所有客体都拥有所有者，访问控制权基于所有者的自由裁量权或决定权。

基于身份的访问控制是自主访问控制DAC的子集。

使用客体访问控制列表ACL实现 DAC 模型。

微软 Windows 系统使用 DAC 模型来管理文件，每个文件或文件夹都有一个 ACL。

14.1.6 非自主访问控制

自主和非自主的区别

自主访问控制：

允许所有者自行修改，也不会影响环境的其他部分。

非自主访问控制：

管理员集中管理，可以实施影响整体环境的变更。

1. 基于角色的访问控制 RBAC

采用基于角色或基于任务的访问控制系统，根据主体角色或分配任务来定义访问客体的能力。

基于角色的访问控制 RBAC 通常通过组来实现。

RBAC 基于角色的访问控制可以预防特权蠕变，有助于加强最小特权原则的实施。

特权蠕变是特权不断增加的意思。

直接分配的权限，很难识别和撤销。

管理员只需要从组中删除用户的账户即可轻松撤销不必要的权限。

管理员按职务描述或工作职能识别角色或组。

用户可以属于多个角色或组。

微软操作系统通过组来实现 RBAC。

DAC 和 RBAC 区别

部署和使用方式不同。

DAC 中，客体的所有者决定权限。

RBAC 中，管理员确定主体权限。

TBAC 和 RBAC

TBAC ：基于任务的访问控制，重点是通过分配的任务来控制访问。

2. 基于规则的访问控制

基于规则的访问控制模型使用一组规则、限制或过滤器，决定系统允许和禁止发生的操作。

显著特征：具有适用于所有主体的全局规则。

常见实例：

防火墙

防火墙包含一个最终规则（即隐式拒绝规则）。

3. 基于属性的访问控制 ABAC

基于属性的访问控制 ABAC 模型是基于规则的访问控制的高级实现方式。

ABAC 模型使用包含多个规则属性的策略。

属性可以是用户、网络和网络上的设备的几乎任何特征。

许多软件定义工程（SDN）应用程序使用 ABAC 模型。

ABAC 模型改进了基于规则的访问控制模型。

基于规则的访问控制模型适用于所有用户，但 ABAC 基于属性的访问控制模型可以更加具体。

4. 强制访问控制 MAC

强制访问控制（MAC）模型依赖分类标签的使用。

一个分类标签代表一个安全域或安全领域。

安全域是共享相同安全策略的主体和客体的集合。

用户根据许可级别获得标签，许可级别是一种特权形式。

私营组织经常使用机密/专有、私有、敏感和公开等标签。

MAC模型通常被称为基于格子的模型。

标有“机密”、“私有”、“敏感” 和 “公开”的水平线标识了分类级别的上限。拥有敏感标签的用户可以访问敏感和公开之间的敏感数据。

MAC 强制访问控制模型允许标签标识出更多定义的安全域。不同安全域放在单独的隔离区进行维护，以提供额外的保护层。

用户需要分类标签和安全域标签才可以访问指定数据，例如：Foil 区域的数据需要 Foil 和 机密标签。

敏感数据（公开边界和敏感边界之间）没有任何其他标签，可以授权拥有敏感标签的用户访问具有敏感标签的任何数据。

组织员工识别标签，定义标签含义和获得标签的需求，管理员分配给主体和客体。

MAC 强制访问控制模型 中使用隔离区，也就是不同的安全域，有助于加强因需可知原则。

MAC 强制访问控制模型是禁止性的，不是允许性的，遵循隐式拒绝原则。

MAC 模型比 DAC 模型更安全，但 MAC模型不具有灵活性和可扩展性。

安全分类代表敏感度级别，未分类的标签是最不敏感的数据。

MAC 模型的分类环境
分层环境

分层环境将各种分类标签关联到按低安全性、中等安全性、高安全性排序的结果。

高级别允许访问低级别，低级别不允许访问高级别。

分区环境

安全域之间是无关联的，每个域代表单独的隔离区。

混合环境

混合环境结合了分层和分区的概念，每个分层级别可能包含多个子分区，子分区之间是相互隔离的。

实现了访问的精细化控制，随着规模增大，混合 MAC 环境越来越难以管理。

14.2.1 互联网上实现 SSO

1. XML

可扩展标记语言（XML）包含标记，从而描述所需要的数据。

依托 XML 格式导入导出数据，使 XML 称为交换信息的通用语言。

云服务商使用 XML 或 基于 XML 的其他语言。

2. SAML

安全断言标记语言（SAML）是一种基于 XML的开放标准，通常用于在联邦组织之间交换身份认证和授权（AA）信息。

SAML 提供支持浏览器访问的 SSO 功能。

结构化信息标准促进组织（OASIS）。

2005 年 SAML 2.0 被纳入 OASIS 标准。

SAML 2.0 规范使用三个实体：委托人、服务提供商和身份提供者。

委托人或用户代理：用户

服务提供商（SP）：站点

身份提供者（IdP）：持有身份认证和授权信息的第三方

身份提供者 IdP 发送三种类型 XML 消息（称为声明）

认证声明：

证明用户代理提供了正确的身份凭证，标识身份认证方法，并表示用户代理登录时间

授权声明：

表明用户代理是否有权访问所请求的服务。如果拒绝，会说明原因。

属性声明：

关于用户代理的任何信息。

3. OAuth

OAuth 2.0 ：开放授权，是RFC 维护的一个授权框架。

OAuth 2.0 不向后兼容 OAuth 1.0。

4. OpenID

OpenID 也是一个开放标准。

提供去中心化身份认证，允许用户使用一组凭证登录多个不相关的网站，凭证由第三方服务维护，也就是 OpenID 提供方。

OpenID 标识符在前台交换，易于发现。

5. OIDC

OpenID Connect （OIDC）是使用 OAuth 2.0 授权框架的身份认证层。

OIDC 同时提供身份认证和授权。

OIDC 建立在 OpenID 创建的技术之上，使用 JavaScript 对象表示法（JSON）Web 令牌（JWT），也可以称为 ID 令牌。

JWT 提供身份认证，包含用户个人资料信息。

OIDC 过程大部分在后台。

OAuth 和 OIDC 组合使用，OAuth 提供授权，OIDC 使用 OAuth 框架来授权，并使用 OpenID 技术进行身份认证。

6. 比较 SAML、OAuth、OpenID 和 OIDC

SAML 要点

  • SAML 2.0 是一种基于 XML 的开放标准。
  • OASIS 在 2005 年 将 SAML 2.0 纳入标准。
  • SAML 2.0 使用三个实体，即主体（用户）、服务提供商SP（网络）、身份提供者IdP（持有认证和授权信息的第三方）
  • SAML 可以提供主体的认证、授权和属性信息。

OAuth 要点

  • OAuth 是一个授权框架，而不是一个认证协议。
  • RFC 6749 描述了 OAuth2.0
  • OAuth 使用 API 交换信息
  • 应用程序从身份提供者那里获取访问令牌
  • 应用程序包含用于授权的访问令牌

OpenID 要点

  • OpenID 是一种身份认证标准。
  • OpenID Foundation 维护。
  • OpenID 提供商提供去中心的身份认证。
  • 用户在站点上输入 OpenID 标识符，OpenID 提供商会验证次标识符。

OIDC 要点

  • OIDC 是 使用 OAuth 2.0 的身份认证层。
  • 建立在 OpenID 身份认证标准之上。
  • 提供身份认证和授权
  • 建立在 OpenID 之上，但使用 JSON Web （JWT）令牌

14.2.2 在内部网络上实现 SSO

重点关注 Kerberos 。

远程访问协议：RAUDIUS 和 TACACS+

1. AAA 协议

提供身份认证、授权和计费的协议被称为 AAA 协议。

AAA 协议通过远程访问系统，例如VPN，提供集中式访问控制。

AAA 协议使用身份标识、身份认证、授权和问责制的访问控制要素。

计费要素可以跟踪用户的网络资源使用情况。

2. Kerberos

票证身份认证是一种采用第三方实体来证明身份并提供身份认证的机制。

Kerberos 主要目的是身份认证，签发票证，访问资源时提供票证。

Kerberos 为用户提供单点登录解决方案并保护登录凭证。

Kerberos 第 5 版基于 AES 对称加密协议，实现对称密钥加密（也称为密钥加密）。

Kerberos 使用端到端安全，为身份认证流量提供保密性和完整性，并可以预防窃听和重放攻击。

kerberos 要素

密钥分发中心

密钥分发中心是提供身份认证服务的可信第三方。

使用对称加密技术。

所有客户端和服务器都需要向 KDC 注册。

KDC 维护所有网络成员的密钥。

Kerberos身份认证服务器

身份认证服务器承载 KDC 的功能：票证授予服务（TGS）和身份认证服务（AS）

KDC ？

票证

票证是一种加密消息，可证明主体被授权访问对象。

票证也被称为服务票证（ST）。

主体（如用户）请求访问对象（如文件）的票证，如果已通过认证和授权，Kerberos 会颁发票证。

Kerberos 票证具有特定的生命周期和使用参数。

票证授予的票证

票证授予的票证（TGT）提供证据，证明主体已通过 KDC 的身份认证并且可以请求票证来访问其他对象。

TGT 是经过加密的，包括对称密钥、到期时间和用户 IP 地址。

Kerberos 主体

Kerberos 向 Kerberos 主体发出票证。

Kerberos 主体通常是用户，但也可以是请求票证的任何实体。

Kerberos 领域

一般而言，领域是指由某物控制或统治的区域。

kerberos 领域是由 Kerberos 控制的逻辑区域（如域）。

领域内的主体可以从 Kerberos 请求票证，而 kerberos 可以向领域内的主体发出票证。

Kerberos 的账户数据库存储在目录服务中，例如微软的目录服务 AD。

Kerberos 在客户端、网络服务器和 KDC 之间交换票证。

加密票证可以确保登录凭证、会话密钥和身份认证消息永远不会以明文形式传输。

Kerberos 登录流程工作原理：

1、用户在客户端输入用户名和密码

2、客户端使用 AES 对用户名和密码进行加密并将其传输的 KDC（认证服务器）

3、KDC 根据已知的凭证数据库来验证用户名

4、KDC 生成一个对称密钥，密钥由客户端和 Kerberos 服务器使用

KDC 使用用户口令的哈希值对密钥进行加密，KDC 生成一个加密的附带时间戳的 TGT

5、KDC（认证服务器）将加密的对称密钥、加密的带时间戳的 TGT 传输给客户端

6、客户端存储 TGT 使用，直到过期

客户端使用用户口令的哈希值来解密对称密钥。

客户端的口令永远不会通过网络传输，只是进行验证。也就是通过 KDC 生成的对称密钥。

客户端访问对象资源时，必须通过 Kerberos 服务器获得服务票证，用 TGT 换 服务票证

步骤：

1、客户端将自己的 TGT 发送到 KDC（认证服务器），并请求访问资源

2、KDC 验证 TGT 是否有效并检查其访问控制矩阵，进而验证用户是否具有足够的权限来访问请求资源。

3、KDC （认证服务器）生成服务票证并将其发送到客户端。

4、客户端将服务票证发送到托管资源的服务器或服务

5、托管资源的服务器或服务通过 KDC （认证服务器）验证票证的有效性

6、一旦验证通过，Kerberos 活动结束，客户端和服务端建立会话，进行通信。

Kerberos 是一种通用身份认证机制，适用于本地局域网、远程访问和客户端/服务器资源请求。

Kerberos 存在单点故障风险，KDC 数据泄露，会导致每个系统的密钥都被泄露；KDC 脱机将无法进行主体身份认证。

Kerberos 具有严格的时间要求，默认配置要求所有系统保持 5 分钟以内的时间同步。

网络内部配置时间同步系统，AD域内，一个域控制器（DC）与 NTP 服务器保持时间同步，其他 DC 服务器与第一个 DC 服务器保持同步。

3. RADIUS

远程认证拨入用户服务（RADIUS）集中认证远程访问连接，例如 VPN。

RADIUS 适用于组织拥有多台网络访问服务器（或远程访问服务器）的情况。

用户 -> 网络访问服务器（RADIUS的客户端） -> RADIUS 服务器（身份认证服务器）。

互联网服务器提供商（ISP）使用 RADIUS 实施身份认证。

用户 -> ISP -> RADIUS 服务器

组织也可以使用 RADIUS ，且通常基于位置的安全来实现 RADIUS。

RADIUS 默认使用用户数据报协议（UDP），仅加密口令的交互过程。

RFC 6614 定义可以使用传输控制协议 TCP 上的传输层安全性（TLS）对整个会话进行加密，RADIUS 使用 TCP 2083 端口。

RADIUS 使用 UDP 1812 端口传输 RADIUS 消息，使用 UDP 1813 端口传输 RADIUS 计费信息。

4. TACACS+

Cisco 开发了增强型终端访问控制器访问控制系统（TACACS+）。

TACACS+ 将身份认证、授权和计费划分为独立的进程，可以部署在不同服务器上。

与 RADIUS 一样，TACACS+可以加密所有身份认证信息（口令+会话）。

TACACS+ 使用 TCP 49 端口。可靠性更高。

14.3 了解访问控制攻击

攻击者表示恶意入侵者。

攻击是企图利用系统漏洞并破坏保密性、完整性和可用性的所有尝试。

风险要素

风险

风险是利用漏洞造成损失（例如资产损坏）的可能性。

威胁

威胁是可能导致不良结果的潜在事件，包括攻击者的潜在攻击、自然灾害事件、员工的意外行为。

漏洞

漏洞是任意类型的弱点。

弱点可能是硬件或软件的缺陷或限制造成的，也可能源自安全控制的缺乏。

风险管理试图通过实施控制措施或对策来减少或消除漏洞，或减少潜在威胁的影响。

消除风险是不可能或不可取的，专注于降低风险。

14.3.2 特权提升

特权提升是指给予用户的特权超过了其应该有的权限。

本地管理员有本地设备的权限

域管理员有全域的权限

普通用户不应该有和管理员相同的权限

攻击者使用特权提升技术来获得额外的特权。

特权提升分为横向特权提升和垂直特权提升。

横向特权提升为获得更多相同的权限，也可以称为横向移动。

垂直特权提升为获得更高的权限。

托管服务账户的重要考虑因素是确保账户仅具有服务或应用程序所需的特权。

创建新的服务托管账户，给予最小权限。

1. 使用 su 和 sudo 命令

Linux 系统 root 账户，被称为超级用户 （superuser）账户。

su 命令默认切换到 root 账户，并需要输入 root 账户口令。

sudo 命令是超级用户执行（superuser do）。

sudo 组用户切换到 root 权限时不需要 root 账户口令，仅需要用户口令即可。

命令前添加 sudo ，则以 root 身份运行命令，日志记录普通用户使用 sudo 执行的所有命令。

使用 su命令切换到 su 账户，日志记录则记录 su 账户而不是普通用户账户记录活动。

2. 尽量减少 sudo 命令的使用

减少使用 sudo 命令，可以限制风险。

Windows 中也是最好创建服务账户，而不是使用本地系统账户。

3. 口令攻击

口令是最脆弱的身份认证方式。

特权账户的口令被获取，选择重建整个系统，而不是接受风险。

强口令（strong password）有助于防止口令攻击，要求字符类型的组合足够长。

避免使用明文存储口令，使用强哈希函数（SHA-3）生成口令的哈希值，存储口令的哈希值。

注意：相同口令运算时，哈希算法始终产生相同的哈希值。

不可以使用 MD5 等弱哈希函数进行口令的哈希计算。

避免使用默认口令。

4. 字典攻击

字典攻击是指尝试通过使用预定义数据库中的每个可能密码或公共或预期密码列表来发现密码。

一次性构造的密码是以前使用过的密码，但是只更改其中一个字符。

5. 暴力破解攻击

暴力破解攻击是指通过尝试所有可能的字母、数字和符号的组合来找到用户的口令。

混合攻击（hybrid attack）先进行字典攻击，再使用一对一构造的口令来执行暴力破解攻击。

口令越长，包含字符类型越多，抵御暴力破解的能力就越强。

哈希计算口令进行认证优势：

  • 避免口令在网络上明文传输
  • 口令也不会以明文存储

比较分析或反向哈希匹配攻击：

直接计算口令的哈希值，使用哈希值进行比对猜测原始口令。

GPU 正在被用于暴力破解。

长口令需要更长的时间破解。

6. 喷射攻击

喷射攻击（spraying attack）是一种特殊类型的暴力破解攻击，口令固定，变换不同账户名和系统进行尝试。

在线密码攻击中使用喷射攻击，并试图绕过账户锁定的安全控制措施。
