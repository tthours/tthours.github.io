15-域6安全评估与测试-安全评估与测试

安全评估和测试方案实施定期检查，以确保安全控制措施充分到位，并有效执行其指定功能。

15.1 构建安全评估和测试方案

安全评估和测试方案是信息安全团队的基础维护活动。

方案包括测试、评估和审计，目的是定期验证是否有足够的控制措施和是否有效。

15.1.1 安全测试

安全测试旨在验证某项控制措施是否正常运行。

测试包括：自动化扫描、工具辅助的渗透测试、破坏安全性的手动测试。

测试应该定期实施。

审查安全控制措施时，信息安全管理者（information security manager）需要考虑的因素：

  • 安全测试资源的可用性
  • 待测试控制措施所保护的系统以及应用程序的重要性
  • 待测试系统及应用程序所含信息的敏感性
  • 实现控制措施的机制出现技术故障的可能性
  • 危机安全性的控制措施出现错误配置的可能性
  • 系统遭受攻击的风险
  • 控制措施配置变更的频率
  • 技术环境下可能影响控制措施性能的其他变更
  • 开展控制措施测试的难度和时间
  • 测试对正常业务运营造成的影响

安全测试策略：

频繁的自动化测试，少量的手工测试。

需要对安全测试结果进行审查。

15.1.2 安全评估

安全评估是指对系统、应用程序或其他待测试环境的安全性进行全面审查。

安全人员进行风险评估，识别安全漏洞，提出修复建议。

安全评估包括：安全测试工具的使用、对威胁环境、当前和未来风险、目标环境价值的仔细审查。

安全评估成果：向管理层提交评估结果，报告需要以非技术语言描述的评估结果，并往往以提高待测环境安全性的具体建议作为结论。

评估可以内部执行，也可以外部第三方团队进行。

评估包括的组成部分：

  • 规范（specification）是与待审计系统有关的文档。

    规范通常包括政策、规程、要求、规格及设计。

  • 机制是满足规范的控制措施，可以基于硬件、软件和固件。

  • 活动是人员所采取的行动，行动包括执行备份、导致日志文件或审查账户历史记录。

  • 人员是指执行规范、机制及活动的人员

15.1.3 安全审计

安全审计必须由独立审核员执行。

评估和测试的结果仅供内部使用，旨在评估控制措施，着眼于发现潜在的提升空间。

审计是为了向第三方证明控制措施的有效性而进行的评估。

审计员（auditor）为组织的安全控制状态提供一种客观中立的视角。

审计员撰写的报告面向的是董事会、监管或其他第三方。

1. 内部审计

组织内部审计人员执行，适用于组织内部。

审计负责人直接向高管汇报。

2. 外部审计

外部审计公司执行，因为没有利益冲突，所以具有很高的公信力。

四大审计公司：

  • 安永（Ernst & Young）
  • 德勤（Deloitte & Touche）
  • 普华永道（PricewaterhouseCoopers）
  • 毕马威（KPMG）

3. 第三方审计

由另一个组织或以另一个组织的名义进行的审计。例如：监管审计被监管公司。

执行审计的组织挑选审核员，并设计审计范围。

向其他组织提供服务的组织经常被要求进行第三方审计。

SSAE 18 通用标准，审计员使用该标准对服务组织进行评估，服务组织只需要开展一次第三方评估，可以与用户及潜在用户共享最终评估报告。

SSAE 18 和 ISAE 3402 声明通常被称为服务组织控制SOC审计，以三种形式出现

SOC1 声明：评估可能影响财务报告准确性的组织控制措施。

SOC2 声明：评估组织的控制措施，控制措施影响系统安全性（保密性、完整性和可用性）和隐私。

​ SOC2 审计结果是保密的，通常仅根据保密协议对外共享。

SOC3 声明：评估组织的控制措施，控制措施影响系统安全性（保密性、完整性和可用性）和隐私。

​ SOC3 审计结果是公开披露的。

两种不同类型的 SOC 报告

I 类报告

提供的审计员的意见仅仅涉及管理层提供的描述和控制措施的设计适用性

I 类报告仅涵盖特定时间点。

I 类报告视为文件审查，审计员以书面形式检查。

II 类报告

进一步提供了审计员对控制措施运行有效性的意见，也就是说，审计员实际上确认了控制措施是否运行正常。

II 类报告涵盖更长的时间段：至少 6 个月的运行时间。

II 类报告视为传统的审计，审计员检查文档并现场验证控制功能是否正常。

II 类报告比 I 类报告可靠的多。

4. 审计标准

审计或评估应该确保正确实施控制措施来实现目标。

信息和相关技术控制目标（Control Objectives for Information and related Technologies，COBIT）是一种开展审计和评估的通用框架。

COBIT 描述了组织信息系统需要具备的通用要求。

COBIT 框架由 ISACA 负责维护。

ISO 27001 描述了建立信息安全管理系统的标准方法。

ISO 27002 介绍了信息安全控制措施的更多细节。

15.2 开展漏洞评估

漏洞扫描和渗透测试可以识别系统或应用中包含的技术漏洞。

漏洞是系统和安全控制措施中可能被威胁利用的弱点。

15.2.1 漏洞描述

安全内容自动化协议（Security Content Automation Protocol，SCAP）。

SCAP 组件包括：

  • 通用漏洞披露（Common Vulnerabilities and Exposures，CVE）：提供一种描述安全漏洞的命名系统。
  • 通用漏洞评分系统（Common Vulnerability Scoring System，CVSS）提供一种描述安全漏洞严重性的标准化评分系统。
  • 通用配置枚举（Common Configuration Enumeration，CCE）：提供一种系统配置问题的命名系统。
  • 通用平台枚举（Common Platform Enumeration，CPE）提供一种操作系统、应用程序及设备的命名系统。
  • 可扩展配置检查表描述格式（Extensible Configuration Checklist Description Format，XCCDF）：提供一种描述安全检查表的语言。
  • 开放漏洞评估语言（Open Vulnerability and Assessment Language，OVAL）：提供一种描述安全测试过程的语言。

15.2.2 漏洞扫描

1. 网络发现扫描

网络发现扫描运用多种技术来扫描一段 IP 地址，从而探测存在开放网络端口的系统。

TCP SYN 扫描

向目标系统的每个端口发送一个设置 SYN 标志位的数据包。

如果扫描器收到设置 SYN 和 ACK 标志位的响应数据包，表明端口是开放的。

TCP SYN 扫描也被称为 “半开放”（half- open）扫描

TCP Connect 扫描

向远程系统的某个端口创建全连接。

适用于执行扫描的用户没有运行半开放扫描所需权限的情况。

TCP ACK 扫描

发送设置 ACK 标志位的数据包，表明属于某个开放连接。

适用于尝试确定防火墙规则或防火墙方法。

UDP 扫描

使用 UDP协议对远程系统开展扫描以检查存储的 UDP 服务。

不使用三次握手，UDP 是无连接协议。

Xmas 扫描

发送设置 FIN、PSH 及 URG 标志位的数据包

nmap扫描工具

nmap 扫描工具端口状态

Open（开放）：该端口已经开放，意味着端口上运行着可以主动接受连接请求的应用程序

Closed（关闭）：该端口可以访问，防火墙允许访问该端口，但是该端口上没有运行接受连接请求的程序

Filtered（过滤）：防火墙干扰连接尝试，无法确定端口是否开放还是关闭。

2. 网络漏洞扫描

网络漏洞扫描不仅探测开放端口，还会继续探测目标系统或网络，从而确定是否存在已知漏洞。

误报（false positive report）：系统不存在漏洞，但是扫描器报告存在漏洞。

漏报（false negative report）：系统存在漏洞，但是扫描报告不存在漏洞，此类更加危险。

网络扫描器默认执行未经身份认证的扫描，也限制网络扫描器发现问题的能力。

实施经过身份认证的系统扫描是一种有效的方法，可以提高扫描的准确性，减少误报和漏报。

网络扫描器之拥有待测系统的只读权限。

无限漏洞测试

aircrack-ng 是一种无限网络安全评估工具，测试无限网络的加密算法和其他安全参数进行评估。

aircrack-ng 可以与被动监测技术协同工作，识别网络中的流氓设备。

3. Web 应用程序漏洞扫描

Web 漏洞扫描器是专门检测 Web 应用程序中已知漏洞的工具。

自动化技术来探测，通过操作输入和其他参数来识别 Web 漏洞。

扫描器提供扫描结果报告和修复建议。

Web 扫描器只扫描 Web 应用程序。

网络漏洞扫描器不会深入 Web 应用程序内部结构。

Web 应用程序漏洞扫描适用场景

  • 在初次执行 Web 应用程序漏洞扫描时扫描所有应用程序，可以检测到遗留应用程序的问题。
  • 在初次将任意的新应用程序移植到生产环境前执行 扫描
  • 在将代码变更部署到生产环境前扫描所有修改过的应用程序
  • 定期扫描所有应用程序，根据应用程序优先级安排扫描任务。敏感数据交程序频繁扫描。

支付卡行业数据安全标准（Payment Card Industry Data Security Standard，PCI DSS）要求企业至少每年执行一次 Web 应用程序漏洞扫描，或安装专业的 Web 应用程序防火墙。

4. 数据库漏洞扫描

大多数数据库受到防火墙保护，但是 Web 应用程序会提供访问数据的入口，存在利用应用程序攻击数据库的风险。

sqlmap 是一种常用的开源数据库漏洞扫描工具，检测 Web 应用程序的数据库漏洞。

5. 漏洞管理工作流程

组织采用漏洞管理系统，应该形成一套工作流程来管理漏洞。

工作流程步骤

（1）检测：漏洞的初次识别通常是漏洞扫的结果。

（2）验证：一旦扫描器检测到漏洞，管理员应该验证漏洞，判断是否为误报

（3）修复：应该对验证过的漏洞加以修复

修复方式可能为：采用供应商提供的补丁，修改设备配置，执行规避漏洞的折中方法，安全防火墙以及其他措施

工作流程目的：

确保组织能够有条不紊地检测和修复漏洞。

15.2.3 渗透测试

渗透测试实际上是在尝试攻击系统，所有渗透测试会比漏洞测试方法更深入。

漏洞扫描仅探测漏洞的存在，不会进行攻击性行为。

渗透测试会尝试突破安全控制措施，入侵目标，验证漏洞的存在。

渗透测试需要关注的范围比漏洞扫描更广。

渗透测试阶段

  • 规划阶段

    测试范围和参与规则需要达成一致，明确测试是经过授权的。

  • 信息收集和发现阶段

    人工+自动化工具收集目标环境信息。

    侦察系统功能，发现系统开放端口

  • 攻击阶段

    尝试使用手动和自动漏洞利用工具来破坏系统安全

  • 报告阶段

    总结渗透测试结果，并提出改进系统安全的建议

常见自动化漏洞利用渗透工具：Metasploit

渗透测试人员可以是内部，也可以是公司聘请的外部顾问。

渗透测试类型

白盒渗透测试（White-Box Penetration Test）

向攻击者提供详细的目标系统信息。

绕过侦察步骤，缩短攻击时间，提升漏洞发现可能性。

灰盒渗透测试（Gray-Box Penetration Test）

部分知识测试，用于平衡白盒和黑盒渗透测试的优缺点。

当需要黑盒测试结果时，但是由于时间或成本原因，需要一些知识来完成测试时，使用这种方式。

黑盒渗透测试（Black-Box Penetration Test）

攻击之前不会向测试人员透露任何信息。

测试启动前，需要明确指出参与规则，从高级管理层获得充分授权。

入侵模拟攻击BAS

入侵模拟攻击（BAS）平台寻求自动化渗透测试的某些方面。

目的在于将威胁指标注入网络和系统，以触发其他安全控制。

运行良好的安全方案中，检测和预防措施可以立即检测和/或阻止潜在的恶意流量。

15.2.4 合规性检查

组织创建并维护合规计划，记录每项监管义务，并将其映射到满足目标的明确的安全控制措施。

合规性检查验证合规计划列出的所有控制措施是否正常运行并有效满足监管要求。

定期执行合规检查，确保组织合规计划的正常运转。

15.3 测试软件

应用程序包含特征

  • 软件应用程序通常拥有操作系统、硬件和其他资源的访问特权。

  • 软件应用程序往往处理敏感信息。

  • 许多软件应用程序依赖于存储敏感信息的数据库。

  • 软件应用程序是现代企业和核心，执行关键业务功能。

    软件故障可能对业务造成严重破坏。

软件核心设计原则是软件永远不要依赖用户的正确操作，需要预料意外情况，并从容地处理无效输入、不正确排序的活动以及其他意外情况。也就是异常处理。

15.3.1 代码审查与测试

代码审查与测试是软件测试方案最关键的组成部分。

1. 代码审查 code review

代码审查（code review）是软件评估的基础。

代码审查也被称为“同行评审”（peer review），也就是其他开发人员对代码进行审查。

代码审查过程-范根检查法（Fagan inspection）

包含六个步骤

  • 规划
  • 总览
  • 准备
  • 审查
  • 反工
  • 追踪

范根检查已经规定好每个阶段进入和退出的标准。

必须先满足标准，才可以进入流程的下一阶段。

范根检查级别的审查只出现在严格受限的研发环境，也就是代码缺陷会造成灾难性危害。

代码审查方法-同行评审方法

大多数组织采用稍微宽松的代码审查流程

  • 开发人员在会议上与一个或多个其他团队成员走查（walk through）代码
  • 高级开发人员执行手动代码审查，在将代码移植到生产环境之前签署所有代码
  • 在将代码移植到生产环境之前使用自动化代码审查工具检测常见的缺陷

2. 静态测试 SAST

静态应用程序安全测试（SAST）在不运行软件的情况下，通过分析源代码或编译后的应用程序来评估软件的安全性。

自动化检测工具，检测常见的软件缺陷：缓冲区溢出

静态分析工具运用于软件设计、构建及测试过程。

3. 动态测试 DAST

动态应用程序安全测试（DAST）在软件运行环境下检测软件的安全性。

如果无法接触到软件底层代码，也就是部署他人开发的软件，动态应用程序安全测试是唯一的选择。

常见动态测试示例：Web 应用程序漏洞扫描。

生产环境中，动态测试务必小心开展，以避免服务的意外中断。

动态测试可以使用模拟事务（synthetic transaction）

模拟事务：

一些脚本化的事务用户以及预期结果，关注实际结果是否和预期结果一致。

IAST

交互式安全测试对运行时行为、应用程序性能、流量、框架、组件进行实时分析

RASP

运行时应用程序自我保护是一种在服务区上运行的工具，可以拦截应用程序的调用，并验证数据请求。

道德披露

漏洞私下告知供应商，让供应商在公布之前修复漏洞。

4. 模糊测试

模糊测试是一种特殊的动态测试技术，向软件提供许多不同类型的输入来测试其边界，以发现之前未检测到的缺陷。

突变（mutation 或 dumb）模糊测试

从软件实际操作获取输入值，然后操纵或改变输入值来生产模糊输入。

预生成（智能）模糊测试

设计数据模型，并基于对软件所用数据类型的理解创新的模糊输入。

模糊测试不能完全覆盖程序的代码，一般仅限于检测不涉及复杂业务逻辑的简单漏洞。

模糊测试有助于执行测试覆盖率分析，进而确定测试范围是否完整。

15.3.2 接口测试

接口测试依据接口设计规范评估模块的性能，以确保模块在开发工作完后可以协同工作。

需要测试的接口有三种：

应用编程接口（API）：为代码模块之间的交互提供统一的方法，并以 Web 服务的形式向外发布。

确保 API 实施了所有安全要求。

用户界面（UI）：包括图形用户界面（GUI）和命令行界面。

验证用户界面是否正常工作。

物理接口：存在于操作机械装置、逻辑控制器或其他物理设备的一些应用程序。

谨慎测试物理接口，避免物理接口失效，带来一些潜在的危害。

15.3.3 误用案例测试

一些应用程序会以清晰的示例展示软件用户可能错误使用该应用程序的方法。

使用一种被称为误用案例测试或滥用案例测试的过程来评估软件是否存在已知风险相关的漏洞。

测试人员先列举已知误用案例，手工活自动化攻击的方法利用案例来测试应用程序。

15.3.4 测试覆盖率分析

测试不可能覆盖软件的所有部分。

通常进行测试覆盖率分析，以估计对新软件的测试程度。

测试覆盖率 = 已测用例的数量/全部用户的数量

测试覆盖率五个常见标准

  • 分支覆盖率：在所有 if 和 else 条件下，是否已执行每个 if 语句
  • 条件覆盖率：在所有输入集合下，是否已经测试代码中的每个逻辑？
  • 函数覆盖率：是否已调用代码中的每个函数并返回结果？
  • 循环覆盖率：在导致代码执行多次、一次或零次的条件下，是否已执行代码中的每个循环？
  • 语句覆盖率：测试期间，是否已执行所有代码？

15.3.5 网络监测

安全人员也参与网站的持续监控。

网络检测分为 2 类

被动监测

在流量穿越网络或抵达服务器的过程中，捕获和分析发送到网站的实际网络流量。

被动监测提供真实的监测数据。

真实用户监控（real user monitoring）是被动监测的一种变体。

综合监测（或主动监测）

向网站发起伪造的事务活动，从而评估网站性能。

可以是简单的请求一个页面计算响应时间，也可以是执行复杂的脚步来确认事务活动的结果。

被动监测和主动监测相互配合使用。

被动监测需要用户出现问题起作用，擅长解决用户识别的问题。

综合监测可能忽略真实用户问题，但是可以在问题发现之前进行检测。

15.4 实施安全管理流程

信息安全方案：实施评估和测试 + 各种管理流程。

管理流程目的是监督信息安全计划是否有效运行。

管理流程为安全评估提供一个关键的反馈环路

安全管理审查包括：日志审查、账户管理、备份验证、关键性能和风险指标。

每项审查都遵循一个标准化流程，审查完成后应该获得管理者确认。

15.4.1 日志审查

存储日志数据

执行自动化及人工日志审查

安全信息和事件管理（SIEM）工具包，使大量日志审查的常规工作实现自动化。

SIEM 工具包利用 syslog 功能来获取信息。

日志系统利用网络时间协议NTP 确保向 SIEM 发送日志记录系统和 SIEM 的时钟是同步的。

信息安全管理者应该定期进行日志审查，特别是对于敏感功能，确保特权用户不会滥用职权。

网络流 NetFlow 日志有助于调查安全事件，提供系统连接和传输数据量的记录。

15.4.2 账户管理

账户管理审查确保用户仅仅保留被授予的权限，且为发生未授权的修改。

执行账户管理审查的一种方式是对所有账户进行全面审查。由于时间的问题，该方法仅仅适用于特权账户。

账户管理审查通用示例

（1）管理人员要求系统管理员提供特权用户及其特殊权限的列表，管理人员可以检索列表，以避免篡改

（2）管理人员要求特权审批机构提供特权用户及其特殊权限的列表

（3）特权审批机构提供的列表和系统管理员提供的列表进行比对，确保只有授权用户保留权限。

流程还包括：

  • 确认已解雇用户未保留对系统的访问权限
  • 检查特定账户的书面记录及其任务

组织如果没有时间执行全部流程，可以改用抽样方式，抽样方式仅在随机情况下才是有效的。

随机抽取部分账户验证。

自动执行账户审核流程，身份和访问管理（IAM）供应商可以提供账户审查工作流程，提示管理员开展审查，维护账户文档并提供审计踪迹（audit trail）表明审计的完成。

15.4.3 灾难恢复和业务连续性

一致性的备份方案是业务连续性和灾难恢复计划工作的重要组成部分。

管理人员应该定期检查备份结果，确保流程有效运行并满足组织的数据保护需求。

一致性备份方案可能包括审查日志、审查哈希值和请求系统或文件的实际恢复。

对灾难恢复和业务连续性控制措施定期测试。

15.4.4 培训和意识

培训和意识目的在于努力使员工了解当前的威胁，向员工提供最佳实践，从而保护系统抵御攻击。

从职前培训开始。

根据个人的角色量身定制。

全年定期开展培训和意识提升工作。

使用网络钓鱼模拟来评估安全意识方案的有效性。

15.4.5 关键绩效和风险指标

安全管理人员应该持续监测关键绩效和风险指标。

监测具体指标因组织而异，可能包括如下内容：

  • 遗留漏洞的数量
  • 修复漏洞的耗时
  • 漏洞/缺陷重现
  • 被盗用账户的数量
  • 在代码移植到生产环境前检测到的软件缺陷数量
  • 重复审计的结果
  • 尝试访问已知恶意站点的用户

展示指标的值随时间的变化。定期查看。
