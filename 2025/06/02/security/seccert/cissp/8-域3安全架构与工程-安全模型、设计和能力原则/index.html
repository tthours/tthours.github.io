8-域3安全架构与工程-安全模型、设计和能力原则

8.1 安全设计原则

系统开发的每个阶段都应该考虑安全问题。

编程人员、开发人员、工程师都需要在各自阶段构建安全体系。

从开发项目早期阶段就周密考虑安全的影响。

8.1.1 客体和主体

主体

主体（subject）是发出资源访问请求的主动实体。

主体可以是用户、进程、程序、计算机或机构。

客体

客体（object）是主体想要访问的被动实体。

客体可以是资源（文件、打印机）、用户、进程、程序、计算机或机构。

访问

访问是主体与客体之间的一种关系，可能包括读、写、更改、删除、打印、移动、备份等操作或活动。

主体和客户不是固定的，在一次事件中充当主体的，可能在下一次事件中会充当客体。

信任传递

信任传递（transitive trust）概念为：A 信任 B ，B 信任 C，那么 A 信任 C

严重的安全问题，可以绕过约束和限制。

8.1.2 封闭系统和开放系统

封闭系统

封闭系统 close system

与同一制造商小范围系统协作

标准专有，不会公开

封闭系统更难与相异的系统集成，导致封闭系统更安全

开放系统

开放系统 open system

使用公认的行业标准

更容易和其他厂商集成

采用标准通信组件，存在更多入口和攻击手段。

更受欢迎

更广泛的部署

安全性更依赖于安全和防御性编程实践规范，也更依赖于周密的深度防御部署战略。

API

应用编程接口（application programming interface，API）

允许在计算元素（如应用程序、服务、网络、固件、硬件等）之间进行的一组定义明确的交互。

API 定义了可以提出的请求类型、提出请求的确切方法、交换的数据形式以及其他要求。

API 使计算元素之间的互相操作成为可能。

开源和闭源

开源 open source

源代码和其他内部逻辑向公众公开的解决方案。

公开审查和评议改进

闭源 closed source

源代码和其他内部逻辑对公众保密的解决方案。

供应商和编程人员改进

均可有价出售或免费提供，商用一般是闭源。

闭源程序可以是开放系统、也可以是封闭系统

开源程序可以是开放系统、也可以是封闭系统

8.1.3 默认安全配置

默认暴政（the tyranny of the default）

永远不要假设任何产品的默认安全配置是安全的。

认为默认安全配置是糟糕的，逐项检查优化。

开发人员有责任为产品的每个配置选项提供详细说明。

8.1.4 失效安全

应用程序安全实效（fail securely）

故障管理的第一类措施是程序错误处理，也叫异常处理（exception handling）

try catch 语句，目的是迅速处理错误。

软失效（fail-soft）

失效安全（fail-secure )

失效保障（fail-safe）

失效打开（fail-open）

失效关闭（fail-closed）

物理世界：

实体优先考虑保护人，但在一些情况下，资产却是优先于人得到保护的。

数字世界：

实体专注于保护资产，保护的类型因CIA三元素不同。

软失效：

允许系统在一个组件发生故障后继续运行。

也就是部分故障。

失效保障：

如果产品会影响物理世界，则人的生命和安全是需要优先考虑的因素。

人类保护优先的设计思路被称为失效保障。

发生故障时，系统、设备或产品将还原到一种保护人员健康和安全的状态。

失效安全：

失效安全系统将资产的物理安全置于任何其他考虑因素之上。

例如：银行金库、医学实验室、数据中心等。

物理世界语境中，失效打开和失效保障是同义词，失效关闭与失效安全是同义词。

数字式产品：

安全焦点在数字资产上，从安全三要素（可用性、保密性、完整性）的角度决定优先保护哪个方面。

如果优先保护可用性，故障时，允许保持连接和通信，也就是失效打开。

如果优先考虑保密性和完整性，故障时，必须切断连接或通信，也就是失效安全、失效关闭或失效保障，数据语境中避免使用失效保障。

失效打开状态，牺牲保密性和完整性保护可用性。

失效关闭状态，牺牲可用性来保护完整性和保密性。

物理语境 状态 数字语境
保护人员 失效打开 保护可用性
保护人员 失效保障 保护保密性和完整性
保护资产 失效关闭 保护保密性和完整性
保护资产 失效安全 保护保密性和完整性

8.1.5 保持简单

保持简单 keep it simple

KISS 原则，安全领域表示不要把环境、产品的设计的过于负责，越复杂，越难以保护。

其他类似的概念

不自我重复（DRY）

不在多处重复相同的代码，避免修改时增加难度。

计算极简主义

尽可能少用硬件和软件资源

最低耗能原则

使用适合所需解决方案的最低耗能编程语言

更坏的就是更好的

功能较少的软件可能更安全。

你不需要它（YAGNI）

在真正需要的时候添加能力和功能。

KISS 原则鼓励所有人避免过于复杂的方案，支持简洁、优化的解决方案。

解决方案越简单，越容易保护，越容易排查故障，越容易验证。

8.1.6 零信任 zero trust

零信任（zero trust）指不对机构内部的任何东西自动予以信任。

永不信任，始终验证。

需要执行内部微分网段和严格落实最小特权原则。

零信任被用于各种不同的安全解决方案，包括内部分段防火墙（ISFW）、多因子身份认证（MFA）、身份和访问管理（IAM）以及下一代端点安全。

可信来源是根本不存在。

8.1.7 通过设计保护隐私 PbD

通过设计保护隐私（Privacy by Design，PbD）指 在产品的早期设计阶段就把隐私保护机制集成在产品中。

安全是产品设计和架构的不可分割元素，自开发启动之初，贯穿整个软件开发生命周期，始终都要保持。

PbD 框架基于 七大 基本原则：

  • 主动而非被动，预防而非修补
  • 以默认方式保护隐私
  • 隐私嵌入式设计
  • 所有功能-正和而非零和
  • 端到端安全-全生命周期保护
  • 可见性和透明性
  • 尊重用户隐私

全球隐私标准（GPS），目的在于建立一套通用且协调一致的隐私原则。

8.1.8 信任但要验证

“信任但要验证” 安全模型已不能满足现代安全需求。

使用零信任模型设计。

“信任但要验证”是首先根据初始身份认证流程授予内部“安全”环境访问权，然后依赖各种控制方法。

8.2 用于确保保密性、完整性和可用性的技术

为保证数据的保密性、完整性和可用性（CIA），必须确保可访问数据的所有组件都是安全的且行为端正。

8.2.1 限定

软件设计者借助“进程限定”约束程序的动作。

进程限定（confinement）使进程只能对某些内存位置和资源进行读和写，也就是沙箱（sandboxing）。

沙箱 就是对进程执行最小特权原则。

限定的目的是防止数据被泄露给未经授权的程序、用户和系统。

限定可由操作系统本身执行（进程隔离、内容保护），也可以借助一个限定应用程序或服务（Sandboxie）或虚拟化管理方案（VMware、VirtualBox）

8.2.2 界限

界限可以充当执行限定的手段。

系统上运行的每个进程都被分配了一个授权级别。

授权级别告知操作系统该进程可以执行哪些操作。

进程的界限（bound）由一些限制组成，规定进程可以访问的内存地址和资源。

简单系统可能只有两个授权级别：用户和内核。

更安全的系统会存在物理隔离，而不仅仅是逻辑界限隔离。

8.2.3 隔离

当一个进程因为访问界限的执行而被限定时，就是在隔离状态下运行。

进程隔离可以确保被隔离进程的任何行为只影响与之关联的内存和资源。

隔离（isolation）是用来保护操作环境、操作系统内核以及其他独立应用程序的。

隔离使软失效环境得以实现。

隔离的实现依赖于限制的执行（以设置界限的方式）

限定：确保活跃进程只能访问特定资源（内存等）

界限：对进程的授权限制，使进程只能与规定的资源交互，且只被允许进行特定类型的交互。

隔离：通过界限执行界定的手段。

8.2.4 访问控制

访问控制限制主体对客体的访问。

访问控制可选项：

  • 自主访问控制
  • 基于角色的访问控制
  • 强制性访问控制

8.2.5 信任和保证

可信系统（trusted system）是指所有保护机制协同工作，可以为许多类型的用户处理敏感数据，并使环境保持稳定、安全的系统。

信任是安全机制、功能或能力的体现。

保证（assurance）是指满足安全需求的可信程度。

保证体现了安全机制在提供安全保护方面有多可靠。

保证必须持续保持、即时更新和反复验证。

8.3 理解安全模型的基本概念

模型提供一种把安全策略形式化的手段。

提出一套可供计算机遵守，用于执行构成安全策略的基本安全概念、进程和规程的明确规则。

安全模型

安全模型（security model）为软件设计人员提供了用来衡量自己设计和执行方案的参照物。

安全令牌

安全令牌（token）是与资源关联的独体客体，描述了资源的安全属性。

令牌可在主体请求访问实际客体之间传达有关客体的安全信息。

能力列表

能力列表（capability list）为每个受控客体各保存一列安全属性信息。

没有令牌灵活，也能提供比较快捷的查找。

安全标签

安全标签（security label）

附着在客体上的一个永久部分。

一旦设定，无法更改，提供了防篡改保护措施。

防篡改保护措施令牌和能力列表没有。

8.3.1 可信计算基

可信计算基（trusted computing base，TCB）设计原则是硬件、软件和控制的集合体。

TCB 是完整信息系统的子集。

安全策略和执行方面，TCB 是系统中唯一一个可以信任的部分。

无论遇到什么事情都遵守安全策略，确保系统在所有情况下都行为恰当，是 TCB 组件的责任。

1. 安全边界

系统的安全边界（security perimeter）是一个假想的边界，可将 TCB 与系统的其余部分分隔开。

TCB 如果要与系统的其余部分通信，则必须创建安全信道，也叫可信路径（trusted path）。

可信路径是按严格标准建立的一条信道，可在进行必要通信时不将 TCB暴露给对安全漏洞的恶意利用。

安全边界允许使用可信壳（trusted shell）。

可使主体在不给 TCB 或主体带来风险的情况下进行命令行操作。

2. 参考监视器和内核

TCB 中负责在授权访问请求之前验证对每个资源的访问的部分被称作参考监视器（reference monitor）。

参考监视器处于每个主体和客体之间，用于验证请求主体的凭证是否符合客体访问要求，只对符合系统访问规则的请求授权。

TCB 中用于执行参考监视器功能的组件集合被称为安全内核（security kernel）。

安全内核的目的是启动相关组件来执行参考监视器功能并抵御所有已知攻击。

8.3.2 状态机模型

状态机模型（state machine model）描述了一个无论处于什么状态都始终安全的系统。

基于有限状态机（finite state machine，FSM）的计算机科学定义。

FSM 会在给定的一个输入和一个状态的情况下转换到另一个状态并可能产生一个输出。

下一个状态 = F（输入，当前状态）

输出 = F（输入，当前状态）

状态（state）是特定时刻的系统快照。

状态转换：接受输入或输出时发生转换，产生新的状态。

所有状态转换都必须接受评估。

安全状态机（secure state machine）：每个可能的状态转换都产生了另一个安全状态的系统。

安全状态机模型系统总是以安全状态启动，在所有状态中保持一种安全状态，并且只允许主体以符合安全策略的方式访问资源。

安全状态机模型是许多其他安全模型的基础。

8.3.3 信息流模型

信息流模型（information flow model）专注于控制信息的流动。

信息流模型基于状态机模型。

多级模型：在不同安全级别之间防止未经授权、不安全或受限制的信息流。

信息流模型允许所有得到授权的信息流通过，同时阻止所有未经授权的信息流。

信息流表明了一个对象从一个时间点的一种状态到另一时间点的另一状态的转换。

信息流模型还可通过明确排除所有非定义流动路径来解决隐蔽信道问题，也就是白名单机制。

8.3.4 无干扰模型

无干扰模型（noninterference model）大致基于信息流模型。

无干扰模型关注的是较高安全级别主体的动作会对较低安全级别主体的动作产生的影响。

高级别主体的动作不应该影响到低级别主体的动作。

无干扰模型的使用可以提供一种防止木马、后门、rootkit 等恶意程序造成损害的保护手段。

组合理论

组合理论（composition theories）：解释了一个系统的输出和另一个系统的输入有怎样的关系

组合理论类型：

  • 级联（cascading）：一个系统的输入来自另一个系统的输出。
  • 反馈（feedback）：一个系统向另一个系统提供输入，由后者进行角色互换（A 先为 B 提供输入，B 再向 A 提供输入）
  • 连接（hookup）：一个系统在把输入发送给另一个系统的同时还将其发送给了外部实体。

8.3.5 获取-授予模型

获取-授予模型（take-grant model）用一个定向图规定应该怎样把权限从一个主体传递给另一个主体或者从一个主体传递给另一个客体。

获取-授予模型基本有 4 条规则：

  • 获取规则：允许主体获取客体的权限
  • 授予规则：允许主体向客体授予权限
  • 创建规则：允许主体创建新权限
  • 移除规则：允许主体移除自己拥有的权限

获取和授予规则实际上是一种复制功能，类似于继承权限。

模型要点：

使用授予、获取、创建和移除规则后，能够确定何时可以更改系统相关权限，以及什么位置发生了泄漏（无意间分配了权限）。
